# 公式化弹幕规范（基于 DDmkTCCorpus / Butterfly 语料）

> 参考：`开源项目/Butterfly-main`、`开源项目/DDmkTCCorpus-main`  
> 对齐：`app/personae.py` 中的 `REPLY_CONTRACT_ZH` 与 DanmuAI 横向滚动 Overlay 约束  
> 定位：AI 提示词与人工审阅用的**输出公式**，不是 B 站原生「刷屏/多行/置顶」全量复刻。

## 1. 为什么要公式化

DDmkTCCorpus 收录的是 B 站亚文化弹幕（鬼畜、电竞、动漫等），文本极短、口语化、情绪外显。DanmuAI 的真实上屏链路是 **Qt 横向滚动 + 每条 ≤15 字 + 固定 5 条/批**，与语料里大量「重复刷屏、多行 ASCII、超长梗链」并不兼容。

公式化的目标：从语料中抽取**可迁移的句式与情绪槽位**，映射到 DanmuAI 已有契约，而不是把语料原样喂给模型。

## 2. 硬约束（与产品一致）

| 项 | 规则 |
|----|------|
| 输出形态 | 仅 JSON 字符串数组，无 Markdown、无解释 |
| 条数 | 固定 5 条 |
| 分工 | 第 1–2 条：**强相关当前截图**；第 3–5 条：**直播间氛围泛用** |
| 长度 | 每条 ≤15 个汉字（英文模式 ≤40 字符） |
| 去重 | 同批 5 条互不重复；避免与近 30 条 Levenshtein 过高相似 |
| 禁止 | 人身攻击、低俗、政治煽动、仇恨、泄露隐私；语料中的攻击链仅作语言学参考，**不可输出** |

## 3. 语料侧可迁移特征（摘要）

结合 DDmkTCCorpus 字段与样本文本，对 DanmuAI **有用**的特征：

| 特征 | 语料表现 | DanmuAI 用法 |
|------|----------|--------------|
| 极短句 | 2–8 字高频（「妙啊」「老婆」「天才」） | 单条上限 15 字，优先 4–10 字 |
| 即时反应 | 惊叹、疑问、确认 | 槽位 1–2 的主句式 |
| 情绪外显 | 哈哈、卧槽、？？？、！！！ | 标点克制：每条约 0–2 个 |
| 接话/捧场 | 「继续」「懂了」「排面」 | 槽位 3–5 |
| 轻度玩梗 | 梗需画面可锚定 | 仅槽位 1–2，且必须看得懂 |
| 重复刷屏 | 「得得得…」「▓▓▓…」 | **禁止**（Overlay 单条滚动） |
| 多行/ASCII | 大字、表情块、歌词墙 | **禁止** |

情感标注（语料 CSV：0 高兴 / 1 难过 / 2 愤怒 / 3 惊 / 4 负样本）可映射为**情绪槽**，见下节。

## 4. 五槽位公式（核心）

每批按顺序填槽； `{}` 为从截图推断的填空，`|…|` 为择一。

### 槽位 1 — 画面锚点（强相关）

**公式 A · 指认**

```text
{画面主体/动作} + {判断词}
例：这操作有点秀 / 左边那个是啥 / 进度条卡住了
```

**公式 B · 惊叹**

```text
{叹词} + {画面触发点}
例：哇这细节 / 啊？还能这样 / 离谱但真实
```

**公式 C · 疑问/确认**

```text
{是不是/为啥/咋} + {可见现象}
例：这是在加载吗 / 为啥全屏了 / 这界面熟啊
```

### 槽位 2 — 画面延展（强相关，与槽位 1 句式不同）

**公式 D · 细节点名**

```text
{具体 UI/文字/颜色/布局} + {短评}
例：右上角那个按钮 / 字体好小啊 / 配色有点压抑
```

**公式 E · 因果/预测**

```text
{因为/接下来} + {可见线索}
例：感觉要卡了 / 下一步该点这 / 像是要更新了
```

### 槽位 3 — 氛围接话（泛用）

**公式 F · 单点情绪**

```text
|懂了|绝了|妙啊|真实|可以|继续|稳了|有点东西|
（2–4 字为佳，可带 1 个叹号）
```

### 槽位 4 — 氛围互动（泛用）

**公式 G · 观众位**

```text
{我在/大家} + {看/等/跟}
例：我先看着 / 弹幕跟上 / 等等看结果
```

**公式 H · 轻梗占位**（无画面锚点时禁用具体梗）

```text
|前方高能|经典再现|有点节目效果|这很直播|
```

### 槽位 5 — 氛围收束（泛用）

**公式 I · 推进**

```text
|下一手见|继续看|别停|再看看|节奏可以|
```

**公式 J · 轻度评价**（非槽位 1–2 的重复）

```text
{整体感受词}
例：今天状态行 / 有点意思 / 还行还行
```

## 5. 情绪 × 槽位速查

| 情绪（语料标签） | 槽位 1–2 倾向 | 槽位 3–5 倾向 |
|------------------|---------------|---------------|
| 惊 (3) | 公式 B/C | 公式 F + 「？」少量 |
| 高兴 (0) | 公式 A + 正向词 | 公式 F/G/I |
| 愤怒 (2) | **不用**语料攻击链；改为公式 C 中性疑问 | 公式 J 轻度 |
| 难过 (1) | 少用；画面确实压抑时用公式 D | 公式 I 收束 |
| 负样本 (4) | **不模仿** | 用公式 F/G 中性泛用 |

## 6. 与现有人格的叠加方式

`app/personae.py` 中 `ensure_reply_contract()` 会在 System Prompt 前部强制附加 JSON 契约。公式化用法：

1. **保留**蓝色契约段（5 条 / 2+3 / 长度 / JSON）。
2. **在自定义风格段**追加一句，例如：

```text
风格：B站直播口语。槽位1–2用公式A–E填空；槽位3–5用公式F–J。禁止重复字刷屏、禁止多行、禁止攻击性梗。
```

3. 可与现有人格组合，例如「搞笑玩梗型 + 公式槽位」、「专业分析型 + 公式 A/D」。

## 7. 标准输出示例

**场景：截图显示代码编辑器报错**

```json
["这报错红一片", "行号对不上啊", "妙啊", "我先盯着看", "下一手见"]
```

**场景：截图显示游戏团战**

```json
["左边开团了？", "这血量还能打", "绝了", "弹幕跟上", "继续看"]
```

**场景：截图显示桌面文档**

```json
["这文档字好密", "像在赶 deadline", "真实", "等等看结果", "节奏可以"]
```

## 8. 质检清单（人工或后处理）

- [ ] 恰好 5 条，且为合法 JSON 数组
- [ ] 第 1–2 条离开截图无法成立（换一张图应失效）
- [ ] 第 3–5 条换图仍可成立
- [ ] 每条 ≤15 汉字，无换行
- [ ] 无连续 3 字以上完全相同重复（如「得得得」）
- [ ] 无脏话、无人身攻击、无敏感政治口号
- [ ] 5 条之间用词不高度雷同

## 10. 明确不做

- 不把语料「重复 N 次」搬进 Overlay
- 不用语料训练数据替代当前截图 → API 主链路
- 不把 Danmaku.js 的 live 刷屏语义当作 DanmuAI 生产标准（见 `docs/danmaku_reference.md`）
